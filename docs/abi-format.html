ABI Format

WASM ABI v1 manifest — the JSON the codegen tool reads. Shapes here mirror wasm-abi-v1.schema.json exactly; every example below passes --validate.

Manifest Structure

A WASM ABI manifest is a JSON file that describes the public interface of a Calimero WASM contract: its methods, events, and named types.

interface AbiManifest {
  schema_version: 'wasm-abi/1';       // must be exactly 'wasm-abi/1'
  types: Record<string, AbiTypeDef>;   // named type definitions
  methods: AbiMethod[];           // callable contract methods
  events: AbiEvent[];            // emitted events
  state_root?: string;            // type name backing contract state
  state_version?: number;        // state schema version (migrations)
  migrations?: AbiMigrationEdge[]; // declared migration edges
}

Minimal Example

{
  "schema_version": "wasm-abi/1",
  "types": {
    "SetEntryRequest": {
      "kind": "record",
      "fields": [
        { "name": "key", "type": { "kind": "string" } },
        { "name": "value", "type": { "kind": "string" } }
      ]
    }
  },
  "methods": [
    {
      "name": "get_value",
      "params": [{ "name": "key", "type": { "kind": "string" } }],
      "returns": { "kind": "string" },
      "returns_nullable": true,
      "intent": "read_only"
    },
    {
      "name": "set_value",
      "params": [{ "name": "req", "type": { "$ref": "SetEntryRequest" } }],
      "returns": { "kind": "unit" },
      "intent": "mutating"
    }
  ],
  "events": []
}

Scalar Types

Primitive types are expressed as an object with a kind field:

{ "kind": "bool" }
Boolean
{ "kind": "string" }
UTF-8 string
{ "kind": "unit" }
Void / no return value
{ "kind": "i32" }
Signed 32-bit integer
{ "kind": "i64" }
Signed 64-bit integer
{ "kind": "u32" }
Unsigned 32-bit integer
{ "kind": "u64" }
Unsigned 64-bit integer
{ "kind": "f32" }
32-bit float
{ "kind": "f64" }
64-bit float
{ "kind": "bytes" }
Byte array. Optional "size": N for fixed length; optional free-form "encoding" hint (most SDKs omit it).

Collection Types

list
{ "kind": "list", "items": <type> } — array of items.
map
{ "kind": "map", "key": <type>, "value": <type> } — key-value map.
record
{ "kind": "record", "fields": [<field>, ...] } — inline struct; may carry "inner_type" for a CRDT wrapper.
tuple
{ "kind": "tuple", "elements": [<type>, ...] } — fixed-length heterogeneous tuple. (Accepted by the codegen; core normalizes tuples away and does not emit them.)
$ref
{ "$ref": "TypeName" } — reference to a named type in types.

Any collection may carry an optional "crdt_type" marking it as a CRDT-backed state collection. The 11 supported values:

lww_register
Last-writer-wins register
counter
Additive counter
vector
Growable list
replicated_growable_array
RGA ordered list
authored_vector
Per-author vector
shared_storage
Shared storage region
unordered_map
Hash map
sorted_map
Key-ordered map
authored_map
Per-author map
unordered_set
Hash set
sorted_set
Ordered set

Named Type Definitions

The types map defines reusable named types. A record's fields is an array of { name, type }; a variant's variants is an array of { name, payload? }:

Record (struct)

{
  "kind": "record",
  "fields": [
    { "name": "name", "type": { "kind": "string" } },
    { "name": "count", "type": { "kind": "u32" } }
  ]
}

Variant (enum)

{
  "kind": "variant",
  "variants": [
    { "name": "Pending" },
    { "name": "Done", "payload": { "kind": "string" } }
  ]
}

A third form, { "kind": "alias", "target": <type> }, names an alias for another type.

Methods

{
  "name": "create_post",           // snake_case method name
  "params": [
    { "name": "title", "type": { "kind": "string" } }
  ],
  "returns": { "$ref": "Post" },
  "returns_nullable": false,    // optional: return may be null
  "intent": "mutating",      // optional: read_only | mutating | unspecified
  "xcall_callable": false,     // optional: callable via cross-context call
  "errors": []                // optional: declared typed errors
}

Only name and params are required. Generated TypeScript method: createPost(title: string): Promise<Post>.

Method Intent

A method's optional intent declares whether it reads or writes state — analogous to SQL SELECT vs INSERT/UPDATE:

read_only
Reads state only; no mutation. Emitted for #[app::view] methods.
mutating
Writes / changes state.
unspecified
Not declared; treated as potentially mutating.

In practice only read_only is emitted explicitly; a method with no intent field (equivalent to unspecified) is treated as potentially mutating. The codegen surfaces a declared intent as a JSDoc tag on the generated method (e.g. @intent read_only) — documentation only; it does not yet change how the method is called.

State Versioning & Migrations

When a contract's state schema evolves, the manifest records the current state_version and the migration edges that reach it. Each edge names the migration method and the fromVersion it upgrades:

  "state_version": 3,
  "migrations": [
    { "method": "migrate_v1_to_v2", "fromVersion": 1 },
    { "method": "migrate_v2_to_v3", "fromVersion": 2 }
  ]

Modelled as AbiMigrationEdge { method: string; fromVersion: number }. Both fields are top-level and optional on the manifest.

Events

{
  "name": "PostCreated",
  "payload": { "$ref": "PostCreatedPayload" }
}

An event has a name and an optional payload (a single type reference — typically a $ref to a record, not inline fields). Events are emitted by the contract during execution and received via the SSE subscription. The codegen generates a typed payload interface: PostCreatedEvent.
